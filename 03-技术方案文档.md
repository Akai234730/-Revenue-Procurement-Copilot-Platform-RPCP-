# 03 技术方案文档

## 1. 技术方案目标

本文档用于定义销售采购一体化智能体平台的企业级技术架构、核心组件、数据模型、集成方式、安全治理、部署方案与可运维能力，确保系统达到生产可用、可扩展、可治理、可审计的水准。

## 2. 技术建设原则

1. 平台化优先：统一底座，避免每个场景重复建设。
2. 组件化设计：能力解耦，支持场景快速编排。
3. 人机协同：关键动作必须可确认、可回退、可审计。
4. 数据驱动：所有分析、评分、建议必须建立在数据与证据上。
5. 安全内建：从架构层保证权限、隔离、脱敏、审计。
6. 可观测可治理：能够监控质量、成本、延迟、异常与业务效果。
7. 多模型兼容：支持私有模型、商用模型、本地推理引擎灵活路由。

## 5. 核心组件设计

## 5.1 Agent Orchestrator
职责：
- 接收用户请求或系统触发事件
- 决策调用哪个场景智能体
- 编排工具调用链
- 管理上下文与状态
- 处理失败重试与超时
- 控制模型路由与成本策略

关键设计要点：
- 支持同步与异步任务
- 支持长任务回调
- 支持任务状态机
- 支持人工审批节点

## 5.2 Workflow Engine
建议能力：
- 可视化定义流程
- 条件分支
- 定时任务
- 人工审批节点
- 回调与补偿机制
- SLA 监控

适用场景：
- 线索自动评分流程
- 方案生成审批流程
- 供应商评估复核流程
- 采购询比价归档流程

## 5.3 Tool Framework

- CRM 查询/更新工具

- 文档解析工具

- 规则校验工具
- 导出工具
- 消息通知工具
- 审批提交工具

## 5.4 Prompt Center
关键能力：
- 模板版本管理
- 环境隔离（测试/预发/生产）
- 参数化配置
- 审批发布

## 5.5 Knowledge Service

## 8. 模型架构设计

## 8.1 模型分类
- LLM：用于理解、生成、推理、总结
- Embedding Model：用于向量化
- OCR/Document AI：用于复杂文档识别
- Classifier：用于分类、风险识别、标签预测
- Scoring Model：用于评分与排序

## 8.2 模型路由策略
建议按任务类型路由：
- 简单分类/抽取：轻量模型优先
- 高质量方案生成：强推理模型优先
- 敏感内容：优先私有模型或内网模型
- 大批量离线任务：成本优先模型

## 8.3 模型治理
- 提示词版本化
- 输出质量抽检
- 幻觉问题标注
- 失败样本回流
- 多模型效果评估
- 成本监控

## 9. 评分与决策引擎设计

## 9.1 线索评分引擎
建议采用“规则 + 机器学习 + LLM 解释”混合架构：
- 规则层：硬性条件打分
- ML 层：基于历史转化数据预测转化概率
- LLM 层：生成自然语言解释与建议

输出：
- 总分
- 维度分
- 转化概率
- 推荐动作
- 证据说明

## 9.2 供应商评分引擎
建议维度：
- 价格 20%
- 交付 20%
- 质量 20%
- 服务 10%
- 风险 15%
- 资质 10%
- 战略价值 5%

支持：
- 权重可配置
- 不同行业/品类模板
- 时间衰减因子
- 风险一票否决规则

## 9.3 采购建议引擎
输入：
- 价格因素
- 条款偏差
- 历史履约
- 风险等级
- 品类策略

输出：
- 推荐供应商排序
- 不推荐原因
- 谈判建议
- 替代建议
- 需要人工确认的风险点

## 10.1 集成对象
- CRM：Salesforce、Dynamics、纷享销客、销售易等
- ERP：SAP、Oracle、用友、金蝶等
- SRM：供应商管理平台

## 10.2 集成方式
- REST API
- Webhook
- Message Queue
- CDC 数据同步
- 定时批处理
- SFTP 文件交换

## 10.3 集成原则
- 优先非侵入式集成
- 接口幂等
- 失败可重试
- 全链路审计
- 最小权限访问

## 11. 安全与合规设计

## 11.1 身份与权限
- 企业 SSO
- MFA 可选
- RBAC + ABAC 组合授权
- 数据域隔离
- 知识权限过滤

## 11.2 数据安全
- 传输加密（TLS）
- 存储加密（KMS）
- 敏感字段脱敏
- 文档水印
- 秘钥托管
- 高敏数据不出域

## 11.3 AI 风险控制
- 敏感词过滤
- 违规内容拦截
- Prompt Injection 防护
- 越权检索防护
- 高风险建议强制人工复核
- 外发文档合规审查

## 11.4 审计留痕
需记录：
- 谁在什么时间发起什么任务
- 使用了哪些知识来源
- 调用了哪些模型与工具
- 输出了什么结论
- 谁做了人工修改或审批

## 12. 高可用与稳定性设计

### 12.1 高可用策略
- 服务多副本部署
- 网关冗余
- 异步任务队列
- 缓存加速
- 限流熔断
- 模型调用超时与降级

### 12.2 降级策略
- 模型不可用时返回模板化建议
- 检索失败时返回有限上下文提示
- 复杂任务改为异步处理
- 外部系统不可用时排队补偿

## 14. 部署方案建议

## 14.1 部署模式
企业级建议优先：
- 私有化部署
- 专有云部署
- 混合云部署

不建议首期采用完全公网 SaaS 直连敏感企业数据。

## 14.2 环境规划
- DEV 开发环境
- SIT 集成测试环境
- UAT 用户验收环境
- PROD 生产环境

## 14.3 核心中间件建议
- API Gateway：Kong/Apigee/Nginx
- Workflow：Temporal/Camunda
- MQ：Kafka/RabbitMQ
- Cache：Redis
- DB：PostgreSQL/MySQL
- Vector DB：Milvus/pgvector/Weaviate
- Object Storage：MinIO/S3
- Monitoring：Prometheus + Grafana
- Logs：ELK/OpenSearch

## 16. 推荐技术栈

后端建议：
- Java/Spring Boot 或 Python/FastAPI
- 规则引擎可选 Drools/自研规则中心
- 工作流引擎 Temporal 或 Camunda

前端建议：
- React + TypeScript + 企业级组件库
- 工作台 + 配置台 + 管理驾驶舱统一建设

AI 相关建议：
- LLM 网关统一封装
- 向量检索服务独立部署
- 文档解析与 OCR 独立服务化
- 评估平台独立建设

## 17. 技术实施关键成功因素

1. 先打通数据与知识，而不是先追求“最强模型”。
2. 先完成关键业务闭环，而不是做孤立 Demo。
3. 强化评分、建议、推荐的可解释性。
4. 全面建设评估体系与运维体系。
5. 对高风险场景坚持人工确认。
6. 用平台思维建设，避免四个模块各自为战。

## 18. 结论

本项目的技术关键不是单一 AI 能力，而是工程化能力：
- 让 AI 能稳定接入真实业务系统
- 让 AI 的结果可信、可解释、可审计
- 让业务可持续迭代优化

按照本技术方案实施，可满足中大型企业对可上线、可扩展、可治理 AI 平台的核心要求。
