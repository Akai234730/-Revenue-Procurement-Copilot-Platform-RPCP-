# 08 部署架构与运维手册

## 1. 文档目标

本文档用于定义销售采购一体化智能体平台的生产部署架构、环境规划、容量建议、发布规范、监控告警、故障处理、备份恢复、安全运维与持续运营机制，确保系统达到企业级上线与稳定运营标准。

## 2. 部署原则

- 生产级部署，不采用单机演示架构
- 核心服务高可用
- 模型服务与业务服务分层部署
- 数据与知识分级保护
- 灰度发布与快速回滚
- 全链路可观测
- 统一配置、统一日志、统一监控、统一密钥管理

## 3. 环境规划

建议四套环境：
- DEV：开发环境
- SIT：系统集成测试环境
- UAT：用户验收环境
- PROD：生产环境

可根据企业规模增加：
- PERF：性能测试环境
- DR：灾备环境

## 4. 逻辑部署架构

推荐逻辑分层如下：

### 4.1 接入层
- 负载均衡
- WAF
- API Gateway
- SSO 接入

### 4.2 应用服务层
- 用户与权限服务
- 线索智能体服务
- 方案生成服务
- 供应商评估服务
- 采购询比价服务
- 知识服务
- 工作流服务
- 审计服务
- 通知服务

### 4.3 AI 服务层
- LLM Gateway
- Prompt Service
- RAG 检索服务
- OCR/文档解析服务
- Embedding 服务
- Rerank 服务
- 评分服务

### 4.4 数据层
- 事务数据库
- 向量数据库
- Redis 缓存
- 对象存储
- 日志存储
- 指标存储

### 4.5 运维与安全层
- Prometheus/Grafana
- ELK/OpenSearch
- Alertmanager
- KMS/Vault
- CI/CD 平台
- 镜像仓库

## 5. 物理部署建议

### 5.1 推荐部署方式
优先顺序建议：
1. 企业私有云 Kubernetes 集群
2. 专有云 Kubernetes 集群
3. 混合云模式

### 5.2 节点规划建议
生产环境建议分池：
- 通用应用节点池
- AI 推理节点池
- 数据服务节点池
- 运维工具节点池

### 5.3 GPU 资源建议
如果采用私有大模型或文档解析推理，建议独立 GPU 节点池，避免与业务服务混部。

## 6. 容量规划建议

## 6.1 关键容量因素
- 日均活跃用户数
- 峰值并发任务数
- 日均上传文档量
- 文档平均页数
- RAG 查询频率
- 长文本生成频率
- 报价解析批量任务量

## 6.2 参考容量分级
### 小规模试点
- 100~300 用户
- 峰值并发 20~50
- 文档日上传 100~300 份

### 中规模正式上线
- 500~1500 用户
- 峰值并发 100~300
- 文档日上传 500~2000 份

### 集团级扩展
- 3000+ 用户
- 峰值并发 500+
- 文档日上传 5000+ 份

## 7. 数据存储规划

### 7.1 事务数据库
用途：
- 用户权限
- 业务主数据
- 流程状态
- 审计元数据

建议：
- 主从或高可用集群
- 定期备份
- 慢 SQL 监控

### 7.2 对象存储
用途：
- 原始文档
- 生成文档
- 附件
- 日志归档

建议：
- 多副本
- 生命周期管理
- 访问控制分级

### 7.3 向量数据库
用途：
- 知识检索
- 文档向量索引

建议：
- 独立资源池
- 分库分集合按领域管理
- 备份索引元数据

## 8. 网络与安全部署

### 8.1 网络分区建议
- DMZ 区：对外接入层
- 业务服务区
- AI 服务区
- 数据区
- 运维管理区

### 8.2 安全控制建议
- 南北向流量 WAF 保护
- 东西向流量服务网格治理
- 零信任访问控制
- 关键管理入口跳板机保护
- 数据库最小权限访问

## 9. 配置与密钥管理

### 9.1 配置管理
- 环境变量标准化
- 配置中心统一管理
- 禁止敏感配置写死代码

### 9.2 密钥管理
- 模型 API Key 托管在 KMS/Vault
- 数据库密码定期轮换
- 第三方接口密钥分环境管理
- 全量访问审计

## 10. 发布管理规范

## 10.1 发布流程
- 代码合并
- 自动化构建
- 单元测试
- 镜像生成
- 安全扫描
- 部署到 SIT/UAT
- UAT 验收
- 生产灰度发布
- 全量发布

## 10.2 发布策略
建议优先采用：
- 蓝绿发布
- 金丝雀发布
- 灰度发布

### 10.3 回滚策略
- 应用版本回滚
- 配置版本回滚
- Prompt 版本回滚
- 模型路由回滚
- 知识版本回滚

## 11. 监控体系

## 11.1 基础设施监控
- CPU
- 内存
- 磁盘
- 网络
- 节点可用性
- GPU 利用率

## 11.2 应用监控
- QPS
- 接口耗时
- 错误率
- 队列积压
- 工作流超时
- 外部接口成功率

## 11.3 AI 监控
- 模型调用成功率
- 平均生成时长
- Token 消耗
- 单任务成本
- OCR 成功率
- 检索命中率
- 幻觉反馈率

## 11.4 业务监控
- 线索处理量
- 方案生成量
- 供应商评估量
- 询比价任务量
- AI 建议采纳率

## 12. 告警策略

### 12.1 P1 告警
- 平台不可用
- 登录失败率异常
- 核心数据库不可用
- 模型网关全量失败
- 关键数据写回失败

### 12.2 P2 告警
- 核心服务错误率升高
- OCR 服务成功率下降
- 队列堆积超阈值
- 向量检索异常
- 外部系统响应超时

### 12.3 P3 告警
- 单节点资源高负载
- 非核心任务延迟偏高
- 某个场景采纳率异常下降

## 13. 日志规范

### 13.1 日志分类
- 访问日志
- 应用日志
- 审计日志
- 安全日志
- AI 任务日志
- 接口联调用日志

### 13.2 日志字段建议
- timestamp
- trace_id
- request_id
- user_id
- org_id
- module_name
- action_name
- result
- cost_ms
- error_code

## 14. 备份与恢复

## 14.1 备份对象
- 事务数据库
- 对象存储元数据
- 配置中心
- Prompt 配置
- 评分规则
- 知识索引元数据

## 14.2 备份策略
- 数据库每日全量 + 增量备份
- 核心配置每次变更自动备份
- 文档对象存储按策略归档

## 14.3 恢复演练
建议至少每季度执行一次：
- 数据库恢复演练
- 配置回滚演练
- 核心链路切换演练

## 15. 灾备方案

### 15.1 RTO/RPO 建议
- 核心业务服务 RTO：2 小时以内
- 核心数据 RPO：15 分钟以内

### 15.2 灾备模式
- 同城双可用区
- 异地灾备
- 关键数据双副本

## 16. 故障处理手册

## 16.1 常见故障场景
- 模型调用超时
- 文档解析失败率上升
- CRM/ERP/SRM 接口不可用
- 工作流卡住
- 向量检索超时
- 回写失败

## 16.2 处理流程
1. 告警触发
2. 值班人员初步定位
3. 判断影响范围
4. 启动应急预案
5. 执行降级或切换
6. 恢复后复盘

## 17. 运维值班机制

建议机制：
- 工作时间值守
- 非工作时间 on-call
- P1 30 分钟内响应
- P2 2 小时内响应
- P3 工作日处理

## 18. 安全运维要求

- 运维操作双人复核
- 高权限账号审批开通
- 运维命令留痕
- 堡垒机审计
- 漏洞定期扫描
- 镜像安全扫描
- 依赖库漏洞治理

## 19. 上线检查清单

上线前必须确认：
- 服务健康检查通过
- 配置正确
- 密钥正确加载
- 灰度策略配置完成
- 监控告警已生效
- 回滚预案已确认
- 关键负责人待命

## 20. 运营优化建议

- 每周查看资源使用趋势
- 每月优化模型成本与缓存策略
- 每月审查告警噪音
- 每季度执行容量回顾与扩容计划

## 21. 结论

稳定部署与运维体系是企业级 AI 平台的生命线。只有建立完善的部署、发布、监控、告警、备份、恢复与应急机制，平台才能真正承载企业核心业务。
